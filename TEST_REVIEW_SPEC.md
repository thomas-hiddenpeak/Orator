# Orator Test Review Spec

本文件定义 Orator 在完成真实测试后默认使用的人工评审规格。

这里的“人工评审”特指：由 AI 在禁用脚本自动打结论的前提下，实际读取参考文本、timeline 输出、日志与关键片段，吃入上下文后进行逐段比对并给出判断。
它不是把任务转交给人类评审，也不是用 CER、DER 或其他脚本指标直接代替最终结论。

适用范围：
- WebSocket 流式 ASR + diarization 联合测试
- 离线整段 ASR 测试
- 参考文本已知、需要人工判断语义与说话人准确率的回归测试

不依赖脚本自动对比结论。可以读取 JSON、日志、参考文本，但最终结论必须是 AI 基于上下文比对做出的人工语义评审。

## 一、评审输入

每次评审至少读取以下材料：

1. 参考文本
   - 例如：asrTest2Final.txt
2. 模型输出
   - timeline JSON
   - 增量 ASR 事件
   - diarization 轨道
3. 测试元信息
   - 音频时长
   - wall time
   - ASR/diar compute time
   - real-time factor
4. 关键运行日志
   - 是否崩溃
   - 是否有明显异常告警

5. AI 人工比对要求
   - 必须实际读取参考文本与模型输出的关键内容
   - 必须进行逐段或分片的上下文对照
   - 不允许只引用脚本统计值直接下结论
   - 不允许把“建议人工复核”当作默认交付物替代评审

## 二、输出结构

每次测试完成后，默认输出以下 4 部分。

### 1. 测试摘要

必须包含：

| 项目 | 内容 |
| --- | --- |
| 测试类型 | 例如 120s WS 流式联测 |
| 输入音频 | 文件名 / 时长 |
| 参考文本 | 使用的 gold 文本 |
| 运行结果 | 成功 / 失败 / 崩溃 |
| Wall time | 实测耗时 |
| Stream RT | 总体实时因子 |
| ASR RT | ASR 实时因子 |
| Diar RT | 说话人分离实时因子 |
| 主观结论 | 一句话总结 |

### 2. 分段人工评审表

按时间线切成若干自然片段。优先按“说话轮次密集变化点”切，不强制等长。

这一步默认由 AI 自行完成：先读取参考文本与输出内容，再按照内容变化和说话轮次进行分段，而不是要求人类先做预切分。

默认表头如下：

| 时间段 | 参考内容摘要 | 系统输出摘要 | ASR 语义评价 | 说话人评价 | 主要问题 |
| --- | --- | --- | --- | --- | --- |

字段解释：

- 时间段：例如 00:00-00:45
- 参考内容摘要：人工概括这一段谁在说、说了什么
- 系统输出摘要：综合 timeline 中 ASR + diar 的实际表现
- ASR 语义评价：准确 / 基本准确 / 部分失真 / 明显错误
- 说话人评价：准确 / 基本准确 / 混淆轻微 / 混淆明显 / 无法用
- 主要问题：例如漏词、反义、粘连、少说话人、抢话合并、切分滞后

### 3. 两项核心结论

必须分别给出：

#### A. ASR 准确率（实际语义）

按语义保真度人工打分，不按字错误率直接代替人工结论。

建议口径：

- 90%-95%：语义高度一致，少量口头词偏差，不影响事实理解
- 80%-89%：主干语义正确，局部词句失真，但可稳定理解讨论内容
- 70%-79%：大意可懂，关键句存在明显错误或遗漏
- 60%-69%：能看出主题，但细节错误较多
- 60%以下：实际语义不可可靠使用

评估时重点看：

1. 讨论主题是否保住
2. 关键数字、否定词、比例、专有名词是否正确
3. 多人打断时是否严重串句
4. 最终文本是否足以支持真实业务理解

#### B. 说话人分离准确率

按“谁在说、是否切对、是否漏人、是否多人并成一人”人工判断。

建议口径：

- 90%-95%：说话人数、切分、归因都基本正确
- 80%-89%：偶有短句错分，但主轮次可靠
- 70%-79%：大段主讲可用，短句与插话有明显混淆
- 60%-69%：只能粗看轮次，不适合精确归因
- 60%以下：说话人轨道不具业务可用性

评估时重点看：

1. 说话人数是否识别完整
2. 主讲段是否持续稳定
3. 抢话/插话是否被合并
4. 是否存在一个真实说话人被并入别人
5. timeline 中 speaker turn 数量是否显著少于真实轮次

### 4. 最终裁定

最终必须给出下面这种形式的结论：

- ASR 语义准确率：约 X%-Y%，一句话解释原因
- 说话人分离准确率：约 X%-Y%，一句话解释原因
- 是否通过本轮测试：通过 / 有条件通过 / 不通过
- 是否建议进入下一轮优化：是 / 否

## 三、默认判定规则

后续测试默认按以下规则输出结论：

1. 先看稳定性
   - 崩溃、断流、无结果，直接判为不通过
2. 再看实时性
   - 记录 wall time 和 RT，不单独决定“准确率”
3. 再看 ASR 语义
   - 优先级高于字面一致率
4. 再看 diarization
   - 若人数都不对，分数上限不超过 69%
5. 最后给综合业务可用性判断
6. 人工评审执行主体
   - 默认由 AI 完成上下文摄取、逐段比对、结论归纳
   - 人类只在你明确要求时才额外介入复核
